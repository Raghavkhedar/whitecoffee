2 · Role decides the inputs

Before evaluating, the two role groups feed different events and a different working window into the same logic.

Check-in eventoffice_infirst site_in
Check-out eventoffice_outlast site_out
Working windowfixed 10:00 – 18:00planned_hours/{date} (admin-set start/end)
No plan set?N/A (window is fixed)Day skipped unless there's approved leave
offMinutes = how far outside the window the day was:
lateMinutes = max(0, actualIn − windowStart) + earlyMinutes = max(0, windowEnd − actualOut)

3 · The 7 statuses & what separates them

StatusConditionSalary (days)
PresentBoth punches · offMinutes = 0 (on time both ends)1.0
SLBoth punches · offMinutes ≤ 120 (≤ 2h off-window)0.75
Half DayBoth punches · offMinutes > 120 (> 2h off-window)0.5
SLNFExactly one punch (in OR out missing)0.5
PLNo punches · approved leave · PL balance > 01.0
UPLNo punches · approved leave · no balance0
AbsentNo punches · no approved leave-2

The 4 things that differentiate everything, in priority order: ① override/Sunday/ops-no-plan → nothing written · ② number of punches (2 / 1 / 0) · ③ offMinutes threshold (0 / ≤120 / >120) · ④ leave + PL balance.

4 · What the status feeds into

Days NP (Net Present)

present + SL×0.75 + halfDay×0.5 + SLNF×0.5 + PL − absent×2

UPL counts as 0.

Salary & PL balance

Salary = daysNP × salaryRate

PL balance: +1 on the 1st of each month (accrueMonthlyLeave), −1 each PL day used (guarded so re-runs don't double-deduct).

5 · Edge cases worth remembering

  • Admin override wins: docs with markedBy: 'admin' (from regularization approvals) are never overwritten by the nightly run.
  • Sundays are skipped — no status, no penalty.
  • Ops with no plan + no leave: day left unmarked; admin must enter a planned shift.
  • UI vs stored doc: the attendance page computes live status from events, so on skipped days the on-screen value can differ from what's actually saved.
